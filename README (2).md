# Dawiyat Open Incident Command Board

A two-file static dashboard tool, designed for GitHub Pages hosting.

## Files

| File | Purpose |
|---|---|
| `index.html` | **Client / TV dashboard.** Read-only. Loads `data.json` and renders all KPIs, charts, the incident table, and both TV modes. |
| `admin.html` | **Data console.** Upload the raw incident export (`.xlsx`), it parses + aggregates everything in your browser, and lets you download a new `data.json`. |
| `data.json` | The dataset that `index.html` reads. Generated by `admin.html`. |

Nothing runs on a server — everything happens in the browser. There is no database; `data.json` **is** the database.

## Hosting on GitHub Pages

1. Create a repo (e.g. `incident-dashboard`) and push these three files to the root (or a `/docs` folder, whichever you point Pages at).
2. In repo **Settings → Pages**, enable Pages for that branch/folder.
3. Your dashboard is now live at `https://<your-username>.github.io/<repo>/index.html`
   and the data console at `https://<your-username>.github.io/<repo>/admin.html`.

## Updating the data (weekly / daily refresh)

1. Export the incident report from your system — the **raw "Page 1" export** with columns:
   `Number, Created, Resolved, Severity, Service Type, Impact, Category, Region, Owner Of RCA, State, Work notes, ...`
2. Open `admin.html` (locally by double-clicking it, or via the hosted GitHub Pages URL).
3. Set the SLA targets (hours) per priority if you need to change them — defaults are P1=4h, P2=8h, P3=24h, P4=48h.
4. Upload the `.xlsx` file → click **Process File**. Review the summary table (per domain + per region) and any warnings.
5. Click **Download data.json**.
6. Replace `data.json` in the repo with the downloaded file (commit + push, or upload via GitHub's web UI).
7. Refresh `index.html` — it picks up the new `data.json` automatically (no code changes needed).

If `data.json` is missing or fails to load, `index.html` falls back to a built‑in sample dataset and shows an amber **"Sample Data"** badge in the header, so the dashboard never looks broken.

## What's in the dashboard

- **Domain tabs**: All / Core / Access / CCTV / E&M — every KPI, chart, and the incident table re-filter per tab.
- **9 visualizations per tab**: Severity mix, Impact composition, Service type, SLA breach gauge, Backlog age composition, RCA owner queue, Monthly open/backlog trend, Open vs Resolved (24h), and **Incidents by Region**.
- **Incident table**: full detail (INC, domain, region, severity, impact, SLA, age, owner, service, created, description, latest update), contained scroll only — the page itself never scrolls.
- **TV Mode — Charts**: fullscreen, cycles through each visualization every 3s with a bold summary panel, then moves to the next domain tab, looping forever.
- **TV Mode — Overview**: fullscreen, whole dashboard stays visible with a subtle "breathing" zoom; every 5s a global summary window (bold, large numbers) shows the stats for the highlighted visualization, cycling the same way.

## Data mapping reference (raw export → dashboard)

| Dashboard field | Source column(s) | Notes |
|---|---|---|
| Domain | `Category` | CCTV→CCTV, Access Network→Access, Electromechanical→E&M, Core Network→Core. Unmapped categories fall into "Other" (flagged as a warning). |
| Region | `Region` | "NORTHERN BOARDER" is normalized to "Northern Border". New region values are picked up automatically. |
| Severity | `Severity` | Parsed from the `(Px)` pattern, e.g. `"3 - Medium (P3)"` → `P3`. |
| Status | `State` | `In Progress` / `On Hold` → Open. `Resolved` → Resolved. `Canceled` → excluded. |
| SLA Breach | computed | `age > SLA target hours for that priority` (targets editable in `admin.html`). |
| Resolved (24h) | `Resolved` | Counted if `Resolved` timestamp is within the last 24 hours of processing time. |
| Age | `Created` vs now (open) / `Resolved` (resolved) | In days, shown in the table and used for the aging buckets. |
| Latest Update | `Work notes` | First (most recent) entry, trimmed. |

If a future export adds new columns or changes these value sets, tell me and I'll extend the mapping in `admin.html` — the dashboard itself (`index.html`) never needs to change since it only reads `data.json`.
